# ROBOSTOCK — 로봇 제작 재고관리 시스템

PARKIE(주차로봇) · CARRIE(물류로봇) · GOALIE(순찰로봇) 제작을 위한
부품 재고 · BOM · 입출고 · 공수 산정 · 작업 인원 일정 통합 관리 도구입니다.

순수 **HTML / CSS / JavaScript** 로만 동작하며, 빌드 도구·프레임워크가 필요 없습니다.
원래 1개의 단일 HTML 파일이었던 것을 GitHub에서 관리하기 좋게 파일별로 분리한 버전입니다.

주요 기능:

- BOM 관리: 레벨 트리, 상위/하위 부품 계층, 상세 모달, 엑셀 업로드/다운로드
- 입출고 관리: 입고/출고 모달, 직접 입력, 신규 부품 일괄 등록, 사진 첨부
- 공수 산정: 모듈별 계획 공수, 실제 공수 기록, 납기 역산, 호기 범위 관리
- 작업 인원 일정: 월간 달력, 30분 단위 드래그 예약, 지연 사유, 확정 일정 간트 표시
- 사용자 기능: 내 계정 모달, 비밀번호 변경, 본인 활동 로그 확인
- 관리자 기능: 사용자 관리, 활동 로그, 앱 내 개발 로그(changelog) 관리
- 반응형 UI: 모바일·폴드·태블릿·데스크톱 레이아웃 대응

## 실행 방법

별도 빌드가 필요 없습니다. 아래 중 한 가지를 쓰면 됩니다.

**가장 간단한 방법** — `index.html`을 더블클릭해서 브라우저로 엽니다.

**권장 방법(로컬 서버)** — 바코드 카메라/일부 기능은 `file://`보다 `http://`에서 더 안정적입니다.

```bash
# Python 3 이 있으면
python3 -m http.server 5500
# 또는 Node 가 있으면
npx serve .
```

브라우저에서 `http://localhost:5500` 접속.

## 데이터 저장 (localStorage)

이 앱은 입력한 데이터를 **브라우저 localStorage 에 자동 저장**합니다.
새로고침하거나 탭을 닫았다 다시 열어도 데이터가 유지됩니다. (서버 불필요)

- 저장 시점: 페이지를 떠날 때 + 탭이 가려질 때 + 로그인 상태에서 2초마다.
- 저장 대상: 부품/재고(`bom`), 입출고 기록(`ioLogs`), 활동로그, 작업인원·예약·지연,
  확정일정, 목표대수, 호기범위, 공수 실적, 문의, 그리고 계정(`USERS`) 변경분.
- **저장 범위의 한계**: localStorage 는 **"그 브라우저, 그 PC"** 에만 저장됩니다.
  다른 사람/다른 기기와 공유되지 않습니다. 공유가 필요하면 백엔드가 필요합니다.
- **용량 한계**: localStorage 는 약 5MB 입니다. 입출고 인증사진(base64)을 많이
  올리면 한도를 넘을 수 있고, 그 경우 저장이 조용히 실패합니다(앱은 계속 동작).
- **데이터 초기화**: 브라우저 콘솔에서 `resetRobostock()` 을 실행하면
  저장된 데이터를 지우고 처음 상태로 되돌립니다.

## 계정 관리

로그인 기본 계정은 `js/config.accounts.js` 의 `DEFAULT_USERS` 에 있습니다.
실제 런타임 계정 관리는 `js/accounts.js` 가 맡고, 관리자(`admin`)는 앱의
"사용자 관리" 화면에서 계정을 추가/수정/삭제할 수 있습니다. 그 변경분은
localStorage 에 저장됩니다.

일반 사용자는 우측 상단 프로필을 눌러 "내 계정" 모달에서 본인 정보와 활동 로그를 보고,
현재 비밀번호 확인 후 새 비밀번호로 변경할 수 있습니다.

## 개발 로그

앱 안에는 관리자 전용 "개발 로그" 페이지가 있습니다.

- 최신 버전과 변경 이력은 `js/data.js` 의 `changelogData` 배열로 관리합니다.
- 상단 앱 버전 표시는 `changelogData`의 최신 항목을 기준으로 자동 표시됩니다.
- 관리자 화면에서 직접 개발 로그를 추가할 수 있으며, 관련 활동은 `activityLog`에도 기록됩니다.

## 버전 정책

이 프로젝트는 **Semantic Versioning (SemVer)** 을 따릅니다.

- 버전 형식: `vMAJOR.MINOR.PATCH`
- 예시: `v0.1.0`, `v1.0.0`, `v1.2.3`

운영 기준:

- `PATCH` 증가: 버그 수정, 오타 수정, 내부 로직 정리, UI 미세 수정
- `MINOR` 증가: 기존 동작을 깨지 않는 기능 추가
- `MAJOR` 증가: 기존 사용법, 데이터 형식, 설정 방식, 외부 연동 방식이 깨지는 변경

권장 운영 흐름:

- 초기 개발 단계는 `0.x.y` 로 운영
- 첫 안정 배포 시 `v1.0.0`
- 배포 태그는 `v1.2.3` 처럼 `v` 포함
- 사전 배포 버전은 `-beta`, `-rc.1` 같은 접미사 사용

현재 이 저장소는 아직 **배포 전 개발 단계**이므로, 정식 `1.0.0` 이전 버전인 `0.x.y`를 유지합니다.
예: `v0.8.20`

배포 시점이 명확해지면 그때 `v1.0.0` 전환 여부를 결정합니다.

자세한 기준은 `VERSIONING.md`를 참고하세요.

### 계정 비공개 모드 (선택 — 깃허브 공개 저장소용)

> ⚠️ 중요: 비밀번호는 클라이언트 코드라 **배포 시 접속자가 볼 수 있습니다.**
> 아래는 "GitHub 저장소 코드에서 숨기는" 정리 수준이며, 진짜 보안은 서버 인증이 필요합니다.

계정을 GitHub 공개 저장소에 올리고 싶지 않다면:

1. `.gitignore` 에서 `# js/config.accounts.js` 줄의 주석(`#`)을 해제합니다.
2. `js/config.accounts.example.js` 를 복사해 `js/config.accounts.js` 를 직접 만듭니다.

이렇게 하면 계정 파일이 저장소에 올라가지 않습니다.
**단, 그러면 GitHub Pages 등 배포본에는 계정 파일이 없어 로그인이 동작하지 않습니다.**
배포 환경에서 직접 `config.accounts.js` 를 넣어줘야 합니다. (이것이 비공개의 대가입니다.)

기본값은 **계정 파일을 저장소에 포함**하므로, 별도 설정 없이 업로드하면 바로 로그인됩니다.

## 기술적 메모 (수정 시 주의)

- **모든 JS 함수는 전역(global)** 입니다. HTML 안의 `onclick="..."` 같은 인라인
  이벤트 핸들러(약 180개)가 이 전역 함수들을 직접 호출하기 때문입니다.
  → ES Module(`import`/`export`)로 바꾸지 마세요. 인라인 핸들러가 전부 깨집니다.
- **스크립트 로드 순서가 곧 원본 코드 순서**입니다. `index.html`의 `<script defer>`
  순서를 임의로 바꾸지 마세요. `defer` 덕분에 순서가 보장됩니다.
- **CSS는 원본 순서를 그대로 4분할**한 것입니다. `<link>` 순서(reset → layout →
  components → pages → responsive)를 바꾸면 캐스케이드(우선순위)가 달라질 수 있습니다.
- 외부 라이브러리는 **XLSX.js(CDN)** 하나뿐입니다. 현재는 비동기(`async`)로 로드되며,
  실제 사용 시점(엑셀 업로드/다운로드 버튼 클릭 전)까지 준비되면 됩니다.
- 폰트는 Google Fonts CDN을 사용합니다(오프라인 필요 시 `assets/fonts/`로 교체).
- **데이터는 브라우저 localStorage 에 자동 저장**됩니다(위 "데이터 저장" 항목 참고).
  새로고침해도 유지되지만, 저장 위치는 그 브라우저/PC 로컬입니다. 여러 사용자
  공유가 필요하면 백엔드 연동이 필요합니다.
- **로그인 계정 초기 시드는 `js/config.accounts.js`**, 실제 계정 로직은
  `js/accounts.js` 에 분리되어 있고, 둘 다 `data.js` 보다 먼저 로드됩니다.
  이 순서를 바꾸지 마세요(로그인이 깨질 수 있음).
- 모바일 상단 메뉴 자동 순환은 `js/mobile-nav.js` 가 담당합니다. 메뉴 DOM 구조와
  `.sb-nav` 클래스명은 이 스크립트가 참조하므로 변경 시 같이 확인해야 합니다.
